# Core Protocols (核心执行协议)

## 协议 0 — 契约写入模式

含 `[HANDOFF-WRITE]` 时优先执行：
1. 覆写 `agents.md`（严格使用原始内容，禁止增删字段）
2. 读取 `workflow_id` 字段
3. 初始化 `.ai_workflow/runtime_state.json`：
```powershell
$wfId = (Select-String -Path agents.md -Pattern 'workflow_id：\[([^\]]+)\]').Matches[0].Groups[1].Value
$state = @{
    schema_version         = "5.2"
    workflow_id            = $wfId
    execution_status       = "pending"
    audit_status           = "pending"
    uat_status             = "not_started"
    merge_status           = "blocked"
    retry_count            = 0
    verification_result    = ""
    verification_artifact  = ".ai_workflow/test_result.xml"
}
New-Item -Force -Path .ai_workflow -ItemType Directory | Out-Null
$state | ConvertTo-Json -Depth 5 | Set-Content .ai_workflow/runtime_state.json
```
4. 执行 `python verify_workspace.py --anchor-only`
5. 输出 `[STAGE: CONTRACT_WRITTEN] agents.md 已写入，runtime_state.json 已初始化，HMAC 锚定完成。`
6. 自动衔接协议 1

## 协议 0.5 — GitNexus 爆炸半径预查询（强制执行）

在执行任何文件修改前，对 Target_Files 核心文件执行：

1. `impact(target: "<核心函数/类>", direction: "upstream")` — 上游调用方
2. `context(target: "<Target_File>")` — 关联测试文件（从 `test_files` 字段提取）

| 结果 | 处理 |
| :--- | :--- |
| 受影响文件均在 Target_Files 内 | `[GITNEXUS: BLAST_RADIUS_OK]` → 协议 1 |
| 存在 Target_Files 外受影响文件 | `[GITNEXUS: BLAST_RADIUS_OVERFLOW]` → 挂起等待更新契约 |
| 查询失败 | `[GITNEXUS: QUERY_FAILED]` → 降级直接执行协议 1 |

测试路径与 `Verification_Command` 不符时输出 `[WARN: TEST_PATH_MISMATCH]`，执行契约路径，上报供云端下轮修正。

## 协议 1 — 状态摄入与鉴权

```powershell
$state = Get-Content .ai_workflow/runtime_state.json -Raw | ConvertFrom-Json
```

`execution_status` 必须为 `pending`，否则输出 `[HALT: STATE_MISMATCH]` 终止。

执行 `python verify_workspace.py --verify-only`：通过 → `[STAGE: HMAC_OK]`；失败 → `[HALT: HMAC_FAILURE]` 终止。

鉴权通过后输出对 `agents.md` 任务目标与执行清单的理解摘要。

**无显式步骤时的执行模式**：若契约执行清单格式为 GitNexus 探索指令 + 任务目标（无 Step N 列表），则：
1. 执行 GitNexus 探索查询，理解依赖关系与代码结构
2. 依据任务目标和 GitNexus 回传结果，自主规划实现路径
3. 在 THOUGHT 日志中说明规划逻辑，充分发挥个人判断与创意
4. 执行前在控制台输出：`[THOUGHT: 自主规划] <简要说明实现思路>`

## 协议 2 — 分支隔离与精准固化

- `Branch_Name` 已存在 → `[HALT: BRANCH_EXISTS]` 挂起；不存在 → `git checkout -b <Branch_Name>`

- **每完成一个逻辑单元（有或无显式 Step），强制执行：**

  **控制台输出**（允许体现个人风格，但须说明修改逻辑）：
  ```
  [THOUGHT: <简短标题>] 修改 <文件>：<说明为什么这么改，选择了什么方案，覆盖了哪个边界>
  ```

  **追加事件日志**（禁止延迟批量）：
  ```powershell
  $entry = @{
      ts            = (Get-Date -Format "yyyy-MM-ddTHH:mm:ss")
      step          = "<逻辑单元描述>"
      files_touched = @("<文件路径>")
      change_reason = "<修改原因>"
      risk_note     = "<已知边界风险，无则填 none>"
      test_impact   = "<对测试的影响，无则填 none>"
  } | ConvertTo-Json -Compress
  Add-Content -Path .ai_workflow/event_log.jsonl -Value $entry
  ```
  写入失败 → `[WARN: EVENT_LOG_WRITE_FAILED]`，当前逻辑单元挂起，等待人类介入。

- **主动黄牌**：执行过程中发现技术债务（即使不影响当前任务），主动写入 `yellow_cards.json`，无需等待 Node C：
  ```powershell
  # 读取并追加，key 格式：相对路径:行号范围
  $yc = Get-Content .ai_workflow/yellow_cards.json -Raw | ConvertFrom-Json
  $key = "<相对路径:行号范围>"
  $yc.cards.$key = @{issue="<问题描述>";workflow_id="<当前wf_id>";ts=(Get-Date -Format "yyyy-MM-ddTHH:mm:ss")}
  $yc.current_count = ($yc.cards.PSObject.Properties | Measure-Object).Count
  $yc | ConvertTo-Json -Depth 5 | Set-Content .ai_workflow/yellow_cards.json
  ```

- 精准 `git add`（只添加 Target_Files 中的文件），commit message：`feat/fix: <描述> [wf-id: <workflow_id>]`

## 协议 4 — 完工验证与交权

```powershell
.\run_verification.ps1 -Caller executor
```

**验证通过（exit 0）**：确认 `execution_status == done`，输出 `[STAGE: EXECUTION_DONE]`，**主动退出进程**。

**验证失败，自修复次数 < 3**：尝试自修复，输出 `[THOUGHT: SELF-FIX N]`。禁止 skip/注释 assert/mock 被测函数/减少验证项。修复后确认 `git diff` 范围在 Target_Files 内，重新调用 wrapper。

**验证失败，自修复次数 = 3**：设 `execution_status = "failed"`，输出完整报错堆栈，挂起等待人类执行 `/REVIEW-ON`。
