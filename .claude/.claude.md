# Core Protocols (核心执行协议)

## 协议 0 — 契约写入模式 (Contract Write-in Mode)

若启动时接收到的消息内容包含 `[HANDOFF-WRITE]` 字符串，进入契约写入模式，优先于一切其他协议执行：

1. 将指令中携带的完整契约内容**覆写**至根目录 `agents.md`（严格使用原始内容，禁止自行增删任何字段）
2. 从 `agents.md` 读取 `workflow_id` 字段值
3. 初始化 `.ai_workflow/runtime_state.json`（若已存在则覆写）：
```powershell
$wfId = (Select-String -Path agents.md -Pattern 'workflow_id：\[([^\]]+)\]').Matches[0].Groups[1].Value
$state = @{
    schema_version         = "5.2"
    workflow_id            = $wfId
    execution_status       = "pending"
    audit_status           = "pending"
    uat_status             = "not_started"
    merge_status           = "blocked"
    retry_count            = 0
    verification_result    = ""
    verification_artifact  = ".ai_workflow/test_result.xml"
}
New-Item -Force -Path .ai_workflow -ItemType Directory | Out-Null
$state | ConvertTo-Json -Depth 5 | Set-Content .ai_workflow/runtime_state.json
```
4. 执行 `python verify_workspace.py --anchor-only` 对五个 SIGNED 字段做初始锚定
5. 向控制台输出：`[STAGE: CONTRACT_WRITTEN] agents.md 已写入，runtime_state.json 已初始化，HMAC 锚定完成。`
6. 随即自动衔接协议 1

若启动时不含 `[HANDOFF-WRITE]`，跳过本协议，直接执行协议 1。

## 协议 0.5 — GitNexus 爆炸半径预查询（强制执行）

> **双端用途说明**：执行端（本节）在修改文件**前**执行爆炸半径阻断，目的是防止超范围写入；Roo 端在审计**后**执行逻辑越权评分（追加 warning，不单独阻断），两者语义不等价，不可互相替代。
>
> **强制执行**：`.gitnexus/` 已确认安装，本协议在每次协议 1 前强制运行。查询失败（`[GITNEXUS: QUERY_FAILED]`）时才降级跳过，直接执行协议 1。

在执行任何文件修改前，对 `agents.md` 中 `Target_Files` 的每个核心文件执行影响分析：

1. **上游调用方查询**：
   ```
   impact(target: "<Target_File 中的核心函数/类>", direction: "upstream")
   ```

2. **关联测试文件发现**：
   ```
   context(target: "<Target_File>")
   ```
   从返回的 `test_files` 字段提取关联测试路径，输出：
   `[GITNEXUS: TEST_FILES_FOUND] 关联测试文件：<路径列表>`
   若与 `Verification_Command` 中路径不一致 → 输出 `[WARN: TEST_PATH_MISMATCH]`，执行契约路径，上报供云端下轮修正

3. **结果路由**：

| 爆炸半径结果 | 处理方式 |
| :--- | :--- |
| 所有受影响文件均在 `Target_Files` 内 | `[GITNEXUS: BLAST_RADIUS_OK]` 进入协议 1 |
| 存在 Target_Files 外的受影响文件 | `[GITNEXUS: BLAST_RADIUS_OVERFLOW]` 挂起，等待人类更新契约后重新下发 |
| 查询失败 | `[GITNEXUS: QUERY_FAILED]` 降级，进入协议 1 |

## 协议 1 — 状态摄入与鉴权握手 (State Intake)

> **语言执行点**：从本协议开始，所有终端输出必须使用简体中文（信号常量标签本身除外）。

**启动后首要动作**：

```powershell
# 读取四轴状态
$state = Get-Content .ai_workflow/runtime_state.json -Raw | ConvertFrom-Json
```

```xml
<rule id="state-assertion" severity="CRITICAL">
  <check>$state.execution_status 必须为 "pending"</check>
  <on-fail>
    输出 [HALT: STATE_MISMATCH] 当前 execution_status=$($state.execution_status)，
    预期 pending。请检查 runtime_state.json 或重新执行 /HANDOFF。
    终止执行。
  </on-fail>
</rule>
```

**HMAC 鉴权**：执行 `python verify_workspace.py --verify-only`
- 通过 → 输出 `[STAGE: HMAC_OK]` 继续
- 失败 → 输出 `[HALT: HMAC_FAILURE]` 终止

**强制握手反馈**：鉴权通过后，向控制台输出对 `agents.md` 中任务目标与执行清单的理解摘要，防止语义偏移。

## 协议 2 — 分支隔离与精准固化 (Branch Isolation)

- 执行编码前，检查 `Branch_Name` 是否已存在：
  - 已存在 → `[HALT: BRANCH_EXISTS]` 挂起，等待人类确认后继续
  - 不存在 → `git checkout -b <Branch_Name>`

- **每完成一个 Step，强制执行以下两步，顺序不可颠倒**：

  **第一步 — 控制台输出**：
  ```
  [THOUGHT: Step N] 修改 <文件>：<一句话说明为什么这么改，覆盖哪个边界情况>
  ```

  **第二步 — 追加事件日志至 event_log.jsonl**（禁止延迟批量写入）：
  ```powershell
  $entry = @{
      ts           = (Get-Date -Format "yyyy-MM-ddTHH:mm:ss")
      step         = N
      files_touched = @("<修改的文件路径>")
      change_reason = "<为什么这么改>"
      risk_note     = "<已知边界风险>"
      test_impact   = "<对测试的影响，无则填 none>"
  } | ConvertTo-Json -Compress
  Add-Content -Path .ai_workflow/event_log.jsonl -Value $entry
  ```

  写入失败 → 输出 `[WARN: EVENT_LOG_WRITE_FAILED]`，**当前 Step 挂起**，等待人类介入。

- 修改完成后，按 `Target_Files` 精准固化，禁止全量 `git add .`
- commit message 格式：`feat/fix: <描述> [wf-id: <workflow_id>]`

## 协议 4 — 完工验证与交权 (Verification & Handoff)

完成全部 Step 并提交代码后，**统一调用验证 wrapper**：

```powershell
.\run_verification.ps1 -Caller executor
```

wrapper 负责：读取命令 → 追加 `--junitxml` → 执行 → 写入 commit hash → 更新 `runtime_state.json`

**验证通过（wrapper exit 0）**：
```powershell
$state = Get-Content .ai_workflow/runtime_state.json -Raw | ConvertFrom-Json
# execution_status 已由 wrapper 设为 "done"，此处仅验证两个字段
if ($state.execution_status -eq "done" -and $state.verification_result -ne "") {
    Write-Host "[STAGE: EXECUTION_DONE] 代码已提交至分支，验证通过，runtime_state 已更新。等待人类 UAT 或 Node C 审计。"
} else {
    Write-Host "[WARN: STATUS_UPDATE_FAILED]"
    Add-Content .ai_workflow/audit_log.txt "[WARN: STATUS_UPDATE_FAILED] $(Get-Date)"
}
```
随后**主动退出进程**，移交控制权。

**验证失败（wrapper exit 非 0），会话内自修复次数 < 3**（自修复计数不写入 `runtime_state.json`，不占用 `retry_count`）：
- 尝试自修复，输出 `[THOUGHT: SELF-FIX N]`
- 禁止：`@pytest.mark.skip`、注释掉 assert、mock 覆盖被测函数、修改测试文件减少验证项
- 自修复后对比 `git diff` 确认范围在 `Target_Files` 内，重新调用 `.\run_verification.ps1`

**验证失败，会话内自修复次数 = 3**：
```powershell
$state = Get-Content .ai_workflow/runtime_state.json -Raw | ConvertFrom-Json
$state.execution_status = "failed"
$state | ConvertTo-Json -Depth 5 | Set-Content .ai_workflow/runtime_state.json
```
输出完整报错堆栈，挂起进程，等待人类前往云端执行 `/REVIEW-ON`。
